# Plan — Daily Review v2: Dual Dashboard + Adaptive Questions + Activity Memory

## Context

The current `#scene-review` in `ui_mockup.html` (lines 391–428) is a static three-row timeline with hardcoded HR/EDA strings. The hackathon goal is to turn the daily review into the centerpiece of the demo: it must feel personal, data-rich, and "smart."

This plan extends EmoMirror with three new capabilities, all surfaced in a redesigned review scene:

1. **Left panel — Dual rolling dashboards.** Two composite scores plotted across the day: **Cardiovascular Load** (HR mean + HR variability across the epoch + motion) and **Stress** (SCR frequency + EDA tonic + skin-temp drop, gated by low motion). Each row is a **chart + radial gauge pair**: the chart on the left shows a 30-min smoothed curve over 24h with **circled highlight peaks** (contiguous ≥15-min runs above personal_mean + 1.5σ) annotated with time + percent-above-baseline; the radial gauge on the right shows today's average filling a circle whose 100% point is the highest historical day's average for that metric. The number inside the gauge is today's score; a small chip shows day-vs-7d-baseline percent.

2. **Right panel — Adaptive questions.** A bank of ~30 templated questions tagged with `triggers`, `category`, `cooldown`, `specificity`. A weighted scorer picks the top 2–3 with category diversity. Questions reference today's specific anomalies ("your stress peaked 2:14–2:51 PM, +84% above your norm — what was happening?") or fall back to open reflection. Voice UI is mocked with text input + a mic icon for the demo (real Whisper integration is out of scope but architected for).

3. **Activity memory loop.** Answers are parsed by a deterministic regex/keyword extractor into `{date, start, end, activity, signature}` events stored under `data/<user>/activity_log.jsonl`. Tomorrow's review compares new highlight windows against past signatures via cosine distance on normalized features and surfaces guesses ("similar to your Monday run?"). **No live LLM call on the demo path** — the extraction step is not shown on stage, so adding network/API risk has no upside (per peer review). An LLM-based extractor stays as a stub for post-hackathon work but is gated behind an env flag and never runs by default.

Because the sample CSV is only ~16 min and `PF` (SCR frequency) has zero rows, the demo runs against **synthesized 7-day history** generated by a new `seed_history.py`. The matching algorithm is real; the data is constructed so the demo lands. SCR frequency is derived locally from `EA` via simple peak detection, not read from `PF`.

### Peer-review corrections incorporated
- **RMSSD removed.** The original plan proposed deriving RMSSD from `60_000/hr_bpm`, which is scientifically invalid — RMSSD requires raw inter-beat intervals at millisecond precision, and EmotiBit's `HR` tag is already a smoothed BPM estimate. Inverting it gives the average RR for the window, not beat-to-beat variability. We substitute **`hr_std` within the 5-min epoch** (legitimate proxy: a calm body has lower HR variance over short windows; an aroused or fluctuating body has higher).
- **Stress is now low-motion-gated.** High EDA + high HR + high motion = exercise sweat, not psychological stress. Stress score multiplies by a `(1 - motion_norm)` factor so that exertion-driven sweat doesn't masquerade as stress.
- **LLM extractor demoted to optional post-hackathon stub.** Demo path is fully offline.
- **Activity memory vector updated.** Replaced `eda_mean` with `scr_freq` to prevent massive 24-hour baseline drift from breaking cross-day activity matching.

## Files to Create

### Backend

**`emomirror/scoring.py`** — composite score math.
- `compute_cardio_score(epoch_features) -> float` (0–100). Weighted: **0.5 × HR_pct + 0.3 × hr_std_pct + 0.2 × motion_pct**. `HR_pct` is HR percentile vs. personal baseline; `hr_std_pct` is the standard deviation of HR samples within the 5-min epoch (proxy for short-window variability — higher = more cardiovascular work); `motion_pct` is the activity_mean percentile. **No motion subtraction** — for cardio, motion-driven HR is *legitimate cardio load* and should count. Cardio is the "effort" axis.
- `compute_stress_score(epoch_features) -> float` (0–100). Weighted: **0.5 × SCR_freq_pct + 0.3 × EDA_tonic_pct + 0.2 × temp_drop_bonus**, then **multiplied by `low_motion_gate = max(0.3, 1.0 - motion_norm)`**. The gate ensures exercise-driven sweat doesn't read as psychological stress — a 5-min epoch with high motion has its stress score attenuated to 30% of the raw value, while a still-body epoch keeps the full score. Stress is the "psychological arousal at rest" axis.
- `derive_scr_events(eda_series, fs) -> list[dict]` — simple peak detector (rolling-baseline subtraction → threshold at 0.05 µS rise → minimum inter-peak gap 1s). Returns `[{t, amplitude, rise_time}]`. Used to compute SCR frequency since the `PF` tag is empty in real EmotiBit data.
- ~~`rmssd(rr_intervals)`~~ **Removed.** EmotiBit `HR` is a derived smoothed BPM, not raw IBI; computing RMSSD from `60_000/bpm` is mathematically meaningless. Use `hr_std` within the epoch instead (see cardio formula).

**`emomirror/review_engine.py`** — the orchestrator the HTML consumes.
- `build_day_review(date, user_id) -> dict` — reads that day's per-tag CSVs (or synthesized day file), splits into 5-min epochs, calls scoring.py, smooths to 30-min window, runs `detect_highlights`, computes day averages, picks questions. Returns the JSON shape below.
- `detect_highlights(epoch_scores, personal_mean, personal_std) -> list[Highlight]` — scans for contiguous epochs where `score > mean + 1.5σ` AND `duration ≥ 15 min`. Merges adjacent runs ≤ 5min apart. Keeps top 2 per metric, non-overlapping. Returns `[{metric, start, end, peak, pct_above_mean, duration_min}]`.
- `summarize_day(epoch_scores, baseline_7d) -> dict` — `{cardio_avg, cardio_vs_baseline_pct, cardio_peak_historical, stress_avg, stress_vs_baseline_pct, stress_peak_historical, dominant_state}`.
- `pick_questions(day_summary, highlights, recent_activities, question_bank, history) -> list[Question]` — implements the weighted fit-score algorithm (see below). Returns top 3 with category diversity enforced.

**`emomirror/question_bank.py`** — static list of ~30 question templates.
- Each entry: `{id, category, text_template, triggers, specificity, cooldown_days, priority_base}`.
- Categories: `anomaly_probe`, `pattern_reinforce`, `state_callout`, `recovery_probe`, `open_reflection`.
- `text_template` supports placeholder fields like `{peak_time}`, `{pct}`, `{duration}`, `{metric}` filled at render time.
- Triggers are structured DSL: `{"stress_score": ">+1.5σ", "highlight_count": ">=1"}` — evaluated against the day summary by `pick_questions`.

**Question selection algorithm** (in `pick_questions`):
```
for q in bank:
    match = trigger_match_strength(q.triggers, day_summary)   # 0..1
    if match == 0: skip
    spec  = q.specificity                                     # 0..1, hand-tuned
    novel = 1.0 if q.category not in last_3_days else 0.5
    cool  = max(0.2, 1.0 - days_since_last_asked(q) / q.cooldown_days)
    fit   = match * (0.6 + 0.4*spec) * novel * cool * q.priority_base
    score[q] = fit

selected = []
for category in [anomaly, state_callout, open_reflection]:   # forced diversity
    pick highest-fit q in this category, append to selected
return selected[:3]
```

**`emomirror/activity_memory.py`** — typed answer → structured event → matching.
- `log_answer(date, transcript, day_context) -> Event` — **regex/keyword extractor is the primary path**. Parses `{activity, start, end, intensity_hint}` from text using:
  - Activity keyword list: `{run, ran, jog, walk, football, soccer, gym, lift, yoga, meeting, presentation, call, meditate, sleep, nap, eat, lunch, dinner, …}` (~30 entries).
  - Time parser: handles `"2pm"`, `"2-3pm"`, `"from 2 to 3"`, `"this morning"`, `"after lunch"`. Falls back to the linked highlight's window if the user didn't name a time.
  - Returns the day's feature signature for that window attached to the event. Appends as JSONL.
- `match_signature(new_window_features, activity_log, top_k=1) -> list[Match]` — cosine distance on normalized `[hr_mean, hr_std, scr_freq, activity_mean, duration_min]` vector. Returns nearest past activities with similarity ≥ 0.85.
- `signature(window_features) -> np.array` — z-score normalize against population baseline from `config.py`.
- **LLM extractor stub:** `_llm_extract(transcript)` exists in the file but is gated behind `if os.getenv("EMOMIRROR_USE_LLM") == "1"`. Default path is regex. The stub is documented as post-hackathon work — its purpose is to handle phrasings the regex misses (e.g., "I crushed a workout", "got grilled in the standup"). Not on the demo critical path.

**`seed_history.py`** (project root) — generate convincing 7-day history.
- For each of the 7 prior days, sample a "day archetype" (e.g., `hard_workout_morning`, `stressful_meetings`, `restful_weekend`).
- For each archetype, generate 288 epochs (24h × 12) of synthetic feature tuples by sampling around archetype-specific means with realistic intra-day curves (circadian HR rise, post-lunch dip, evening wind-down).
- Run them through `review_engine.build_day_review()` and write the resulting JSON to `data/demo_user/history/<date>.json`.
- Pre-populate `data/demo_user/activity_log.jsonl` with 5–7 plausible past activities, each with a feature signature. **One of them must match today's planned highlight** so the matcher demonstrably finds it.
- Also writes `data/demo_user/user_baseline.json` containing: mean+std of all 7 days' epoch features, **and `peak_day` block** = `{cardio: {value, date}, stress: {value, date}}` recording the highest daily-average score for each metric across the 7 days. The frontend gauge reads these as the 100% reference. To make the demo land, ensure today's stress is **near but below** the historical peak (e.g., today=71, peak=89 → ~80% fill — visually substantial without hitting the ceiling).
- **Run once before demo.** Output is checked into the demo branch as the "locked previous days" the user asked about.

**`build_today_review.py`** (project root) — produces `review.json` for the HTML.
- Reads today's CSV (the existing sample, looped/stretched to 24h via simple time scaling) OR a hand-crafted `today.csv` with one obvious highlight window.
- Calls `review_engine.build_day_review(today, demo_user)`.
- Writes `review.json` next to `ui_mockup.html` so the page can `fetch('./review.json')`.

**Radial gauge semantics (the key new visual):**
- The gauge is a circle with a track arc (full 360°, faint) + a filled arc (colored, capped).
- **100% = the highest historical day's average for that metric** (read from `data/demo_user/user_baseline.json`'s `peak_day` field).
- Today's average fills the arc proportionally: `fill_pct = today_avg / historical_peak_avg`.
- The center text is today's average score (large, tabular-nums).
- A small label below ("of your peak") makes the reference frame explicit.
- A delta chip below the gauge shows day-vs-7d-baseline percent, color-coded: green (≤ -5%), neutral white (±5%), amber (+5 to +20%), red (> +20%).
- Why this works: gives an instant "today vs your worst/heaviest ever" frame. The empty arc is "headroom." The user reads the gauge in <1s without parsing axes.

## Executor Notes (read before implementing)

These are decisions and concrete specifics the executing agent should NOT re-derive — they were made deliberately and the demo depends on them.

### A. Today's data source — use synthesized, not the real CSV

`build_today_review.py` should **synthesize today's per-tag CSVs directly**, not read the existing 16-min sample. Reasoning: the real CSV is too short, has empty `PF`, and stretching timestamps gives unrealistic signal density. Use the same synthesizer machinery as `seed_history.py` with a hand-picked `today_archetype`.

- **Today's archetype:** `morning_workout_then_stressful_afternoon` — produces one cardio highlight ~7:12–7:48 AM and one stress highlight ~2:14–2:51 PM.
- **Output location:** write per-tag CSVs to `data/demo_user/today/{HR,EA,T1,AX,AY,AZ}.csv`. `features.extract_features()` reads from there.
- **Skip `EmotiBitDataParser.exe` entirely for the demo path.** The parser is only needed when ingesting real EmotiBit recordings; the synthesizer writes per-tag format directly.
- A `--use-real-csv` flag on `build_today_review.py` can opt into stretching the existing 2026-04-24 sample (`stretched_ts = original_ts × 86400000 / 988677`) for a future real-data demo. Not on the hackathon path.

### D. Trigger DSL — evaluator spec

Triggers are simple key/value pairs evaluated against `day_summary` and `highlights`. Keep it dumb; don't write a parser library.

- **Numeric comparison:** `">+1.5σ"` → split into `(op, value, unit)` = `(">", 1.5, "σ")`. Evaluate as `summary[key] > 1.5 * summary[key + "_std"]`. For `"%"` unit, evaluate as raw percent.
- **Plain comparison:** `">+15"` → `summary[key] > 15`.
- **Equality / period:** `"afternoon"` matches if `highlight.start` falls in 12:00–17:59. Periods: `morning` 05:00–11:59, `afternoon` 12:00–17:59, `evening` 18:00–23:59.
- **Count comparisons:** `">=1"` against arrays like `stress_highlight` evaluates `len(highlights_for_metric) >= 1`.
- **Always-true:** `"always": true` short-circuits to match strength 1.0.
- `trigger_match_strength` returns the **minimum** of all individual trigger matches (AND semantics). Match strength scales: a highlight 73% above mean produces a stronger match than one 16% above (`min(1.0, pct / 50)`).

### E. `pick_questions` parameter shapes

- `day_summary` — the `summary` dict from `review.json` plus `_std` fields for each metric (computed by `summarize_day` from the 7-day baseline).
- `highlights` — flat list `[{metric, start, end, peak, pct_above_mean, duration_min}, ...]`.
- `recent_activities` — `[{date, activity, signature}, ...]` (last 7 days of `activity_log.jsonl`).
- `history` — `{question_id: last_asked_date}` map. Persisted at `data/demo_user/question_history.json`. Updated each time `pick_questions` runs.

### F. Color and load order

- **Cardio = cyan** (`var(--state-calming)`). **Stress = red** (`var(--state-storm)`). Matches existing avatar state colors.
- **Load sequence in `playAwakening()`:**
  1. `await loadReview()` — fetches `review.json`, falls back to hardcoded sample on error.
  2. `renderDashboards(data)` — builds DOM with gauges in their *empty* state (`stroke-dashoffset = circumference`).
  3. `renderQuestions(data)` — populates question cards but keeps them at `opacity: 0`.
  4. Run the existing intro tweens (h1, h2).
  5. Stagger `.dashboard-row` reveal (0.6s).
  6. Trigger gauge fill tweens (`stroke-dashoffset` + counter, 1.4s).
  7. Stagger `.question-card` reveal (0.5s).
  8. Reveal `.actions` last.
- **Total awakening reveal: ≤ 3.5s** before user can interact.

### G. Synthesizer determinism

Both `seed_history.py` and `build_today_review.py` must be **seeded** (`random.seed(20260425)` and `np.random.seed(20260425)`) so reruns produce identical outputs. The demo depends on the matcher hitting the same Monday-run signature every time.
