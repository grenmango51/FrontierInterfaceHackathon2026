# EmoMirror Biofeedback & Data Quality Audit

**Date**: 2026-04-26
**Topic**: Investigation into "silly data" and flat-line biofeedback scores from EmotiBit hardware.

## 1. Observed Problems
- **Flat-line Feedback**: The "Calm" bar (HP/RSA) in the breathing challenge stayed stuck at the bottom or didn't move during the exhale.
- **Static Daily Review**: Historical Stress/Cardio lines looked like flat baselines despite real physiological events.
- **Inconsistent Comments**: Review feedback felt repetitive ("even") regardless of the session data.

## 2. Root Cause Analysis
An audit of the provided CSV (`2026-04-24_21-29-04-581986.csv`) and the codebase revealed a two-part failure:

### A. Hardware/Data Constraints (Reason 2)
- **EDA Saturation**: The EDA (Stress) signal regularly saturates at `10,000.0` (ADC limit), likely due to poor sensor contact or moisture.
- **HR Jumps**: Heart rate values jump by >20 bpm in <1s, indicating motion artifacts or peak detection errors on the device.

### B. Algorithmic Bottlenecks (Reason 3)
- **Over-Smoothing**: `biosignal.js` was using an 8-beat moving average for HR. This erased the Respiratory Sinus Arrhythmia (RSA) oscillation (the variation between inhale/exhale) that the app intended to measure.
- **Outlier Sensitivity**: `csv_parser.js` used global `min/max` for normalization. A single 10,000.0 spike would "compress" the real data (range 2-10) into a flat line at the bottom of the graph.
- **Fixed Targets**: The challenge engine used a hardcoded 9 bpm RSA target. If a user's noisy signal only oscillated by 2 bpm, the bar would never move.

## 3. Implemented Solutions
The following fixes were applied to handle "silly" data gracefully:

| Component | Change | Purpose |
| :--- | :--- | :--- |
| **`biosignal.js`** | Reduced HR window (8 → 3 beats) | Allows instantaneous HR oscillation (RSA) to be visible to the engine. |
| **`csv_parser.js`** | Robust Percentile Normalization | Uses 5th–95th percentile instead of global min/max; ignores 10,000.0 spikes. |
| **`csv_parser.js`** | Lowered State Thresholds | Makes status labels (Calm, Stressed, Anxious) more dynamic and less "stuck." |
| **`ui_mockup.html`** | Outlier Rejection Layer | Jumps > 15 bpm are ignored to prevent noise from ruining the score. |
| **`ui_mockup.html`** | Adaptive "Auto-Gain" Target | The 9 bpm goal now drifts (min 3, max 15) to match the user's actual signal strength. |
| **`ui_mockup.html`** | Visual Glitch Feedback | Linked signal noise to `avatarState.glitch`. Hardware noise is now an aesthetic "interference" effect. |

## 4. Advice for Future Agents
- **Don't wait for better hardware**: The data from these sensors will always be noisy. Always use **Percentile-based clipping** and **Adaptive targets**.
- **Phase-awareness**: The current engine is "Phase-Aware" (it knows when you are exhaling). Use this to reward *trends* (is HR going down?) rather than absolute values if the data is jittery.
- **SQI (Signal Quality Index)**: If building further, implement a dedicated SQI to freeze scores when motion (`AX/AY/AZ`) is too high.
